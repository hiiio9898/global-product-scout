# fix-scraper — 修复抓取失效

## 触发条件
- 用户报告抓取返回空数据或解析异常
- 抓取到的数据字段缺失（如价格全是 None）
- 网站改版导致选择器失效

## 工作流

### 1. 问题定位
- 确认受影响的平台（亚马逊/其他）
- 确认具体症状：
  - 完全无法抓取（HTTP 错误/超时）
  - 能抓取但数据为空（选择器失效）
  - 部分字段缺失（价格/标题/评分）
- 检查是否为临时问题（网站维护、网络波动）

### 2. 分析目标网站变更
- 访问目标网站当前页面，对比已知选择器
- 检查以下变更点：
  - HTML 结构变化（容器 ID/Class 变更）
  - 产品卡片组件重构
  - 价格格式变化（货币符号、小数点格式）
  - 评分展示方式变化（图标替换文字）
  - 反爬升级（新增验证码、JS 渲染）

### 3. 更新抓取逻辑
在 `src/scraper.py` 中修复：
```python
def scrape_<platform>_best_sellers(url, max_products, use_mock):
    # 更新 CSS 选择器
    # 更新数据解析正则
    # 增强 try/except 容错
```
- 优先保留旧选择器作为 fallback（网站可能 A/B 测试）
- 新增更详细的日志输出，便于下次排查
- 更新对应的 `MOCK_*_PRODUCTS` 数据（如页面结构变化影响字段含义）

### 4. 增强容错
- 对每个字段单独 try/except，避免一个字段失败导致整条记录丢失
- 新增字段缺失计数日志，便于监控
- 设置合理的降级策略（返回部分数据 vs 跳过该产品）

### 5. 测试验证
- 使用 mock 数据确认基础逻辑不受影响
- 如可能，用真实页面 HTML 片段编写单元测试
- 确认 `pytest tests/` 通过
- 确认 Streamlit 界面正常展示

### 6. 交付说明
告知用户：
- 网站变更的具体内容
- 修改了哪些选择器/解析逻辑
- 修复的可靠性评估（是否可能再次失效）
- 建议的后续措施（如增加监控、考虑 API 接入）

## 注意事项
- 修复时保持向后兼容，不删除现有字段
- 优先保证 mock 模式正常，确保开发流程不受影响
- 如网站反爬大幅升级，建议评估是否切换为官方 API
- 不执行与修复无关的其他改动
